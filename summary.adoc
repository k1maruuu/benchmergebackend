== Назначение проекта

BenchMerge Backend — это серверная часть системы для управления конфигурациями LLM, тестирования и анализа результатов. Система поддерживает:

* Эволюционный подбор параметров моделей (evolution workflow)
* Автоматическое тестирование и валидацию результатов
* Интеграцию с внешними сервисами (Dispatcher, Workshop, LaunchPad)
* Файловое JSON-хранилище для персистентности данных
* Real-time обновления через WebSocket/SSE

== Точка входа приложения

Точка входа: `benchmergebackend/app/main.py`

Основные компоненты инициализации:

* `FastAPI` приложение с middleware CORS
* `lifespan` контекст для инициализации хранилища и конфигураций
* Подключение 10 роутеров (health, configs, tests, suites, runs, files, board-layout, gateway, realtime, board)

Ключевые вызовы при старте:

* `initialize_storage_tree()` — создание структуры директорий
* `ensure_all_configs_have_trigger()` — валидация конфигураций
* `ensure_default_collector_config()` — создание дефолтной конфигурации Collector

== API-группы

=== Configs (`/configs`)

Управление конфигурациями LLM с поддержкой семейств и триггеров:

* `GET /configs` — список всех конфигураций
* `POST /configs` — создать конфигурацию
* `GET /configs/{id}` — получить конфигурацию
* `PUT /configs/{id}` — обновить конфигурацию
* `DELETE /configs/{id}` — удалить конфигурацию
* `PATCH /configs/{id}/trigger` — обновить триггер
* `PATCH /configs/{id}/options` — обновить опции
* `POST /configs/{id}/offspring` — создать потомков

=== Tests (`/tests`)

Управление тестовыми заданиями:

* `GET /tests` — список тестов
* `POST /tests` — создать тест
* `GET /tests/{id}` — получить тест
* `PUT /tests/{id}` — обновить тест
* `DELETE /tests/{id}` — удалить тест

=== Suites (`/suites`)

Управление наборами тестов:

* `GET /suites` — список сьютов
* `POST /suites` — создать сьют
* `GET /suites/{id}` — получить сьют
* `PUT /suites/{id}` — обновить сьют
* `DELETE /suites/{id}` — удалить сьют

=== Runs (`/runs`)

Управление запусками тестов:

* `GET /runs` — список запусков
* `POST /runs` — создать и запустить
* `GET /runs/{id}` — получить запуск
* `GET /runs/{id}/results` — результаты запуска
* `POST /runs/{id}/select-best` — выбрать лучший результат
* `POST /collector/dispatch` — запустить Collector

=== Gateway (`/gateway`)

Прокси к внешним сервисам:

* `GET /gateway/connections` — список сервисов
* `GET /gateway/health` — health check
* `POST /gateway/run` — запустить сервис
* `POST /gateway/stop` — остановить сервис
* `POST /gateway/launchpad/set-system-prompt` — установить системный промпт
* `POST /gateway/launchpad/pull` — получить payload нода
* `POST /gateway/launchpad/set-payload` — установить payload

== Schemas

Основные Pydantic схемы:

=== ConfigCreate

Создание конфигурации:

* `config_id` (Optional[str]) — пользовательский ID
* `name` (str) — название
* `model` (str) — модель LLM (по умолчанию `gemma3:4b`)
* `service` (Optional[str]) — сервис Dispatcher
* `node` (Optional[str]) — LaunchPad node
* `system_prompt` (Optional[str]) — системный промпт
* `options` (Dict[str, Any]) — параметры генерации
* `description` (Optional[str]) — описание
* `trigger` (Optional[ConfigTrigger]) — флаги автоматизации

=== ConfigTrigger

Флаги автоматизации:

* `collector` (bool) — активировать Collector trigger

=== RunCreate

Создание запуска:

* `suite_id` (Optional[str]) — ID сьюта
* `selected_test_ids` (Optional[List[str]]) — явный список тестов
* `config_ids` (List[str]) — список ID конфигураций
* `service` (Optional[str]) — сервис Dispatcher

=== ResultSchema

Результат теста:

* `result_id` (str)
* `run_id` (str)
* `config_id` (str)
* `test_id` (str)
* `response_text` (str)
* `score` (float) — оценка 0.0-1.0
* `passed` (bool)
* `latency_ms` (int)

=== Storage сущности

* **Config** — конфигурация LLM с family_id, generation, parent_config_id
* **Family** — семейство конфигураций с parent-child связями
* **Test** — тестовое задание с prompts и checks
* **Suite** — набор тестов
* **Run** — запуск с манифестом, индексом и результатами
* **Result** — результат теста с метриками

== Configs, Tests и Runs

=== Workflow: Evolution

1. Создаётся базовая конфигурация (root config, generation=0)
2. Генерируются потомки с мутациями (`create_offspring`)
3. Запускаются тесты для всех конфигураций
4. Collector анализирует результаты
5. Лучшие конфигурации становятся родителями следующего поколения

=== Mutation Strategy

Мутации опций генерации:

* `temperature` — ±0.1, ±0.2
* `top_p` — ±0.05
* `top_k` — ±5, ±10
* `repeat_penalty` — ±0.1
* `num_ctx` — ±512

=== Workflow: Collector Trigger

Автоматический запуск Collector:

1. Родительская конфигурация имеет `trigger.collector = true`
2. Все потомки завершили выполнение
3. Collector получает путь к результатам
4. Collector анализирует и создаёт новый результат

== Ограничения MVP

1. **Файловое хранилище** — не подходит для высоконагруженных сценариев
2. **Отсутствие транзакций** — нет гарантированной атомарности операций
3. **In-memory WebSocket** — не масштабируется на несколько инстансов
4. **Нет автентификации** — открытое API
5. **Нет rate limiting** — защита от злоупотреблений отсутствует
6. **JSON-only validation** — базовая валидация опций LLM
