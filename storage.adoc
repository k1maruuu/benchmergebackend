= Storage Layer Documentation
:toc: left
:toclevels: 3

Архитектурный слой `storage` отвечает за персистентность данных в проекте Benchmerge Backend. Все данные сохраняются в виде JSON-файлов в файловой системе, что обеспечивает простоту отладки, миграции и резервного копирования.

== Обзор структуры

Модуль `app/storage` состоит из следующих компонентов:

- `json_store.py` — базовые операции с JSON-файлами
- `paths.py` — генерация путей к файлам и директориям
- `config_store.py` — хранение конфигураций и семейств
- `test_store.py` — хранение тестов
- `suite_store.py` — хранения наборов тестов (сьютов)
- `run_store.py` — хранение результатов запусков

== Базовые операции с JSON

Файл: `json_store.py`

Предоставляет низкоуровневые функции для работы с JSON-файлами:

[cols="1,3"]
|===
|Функция |Описание

|`ensure_parent(path: Path)`
|Создаёт родительские директории для указанного пути, если они не существуют.

|`read_json(path: Path, default: Any = None)`
|Читает JSON-файл. Если файл не существует, возвращает значение по умолчанию (`None` или переданное `default`).

|`write_json(path: Path, payload: Any)`
|Записывает объект в JSON-файл с отступами (indent=2) и поддержкой UTF-8. Автоматически создаёт родительские директории.

|`delete_json(path: Path)`
|Удаляет JSON-файл, если он существует.

|`list_json_files(path: Path)`
|Возвращает отсортированный список всех `*.json` файлов в указанной директории. Если директория не существует, возвращает пустой список.
|===

== Управление путями

Файл: `paths.py`

Централизованный модуль для построения путей к файлам и директориям в хранилище. Использует настройку `BENCHFILE_STORAGE_ROOT` из конфигурации приложения.

[cols="1,3"]
|===
|Функция |Описание

|`storage_root()`
|Возвращает корневую директорию хранилища (из `settings.storage_root`).

|`initialize_storage_tree()`
|Создаёт всю необходимую иерархию директорий при запуске приложения. Вызывается при старте для гарантии существования всех папок.

|`configs_items_dir()`, `configs_families_dir()`
|Возвращают пути к директориям `configs/items/` и `configs/families/`.

|`tests_items_dir()`, `tests_suites_dir()`
|Возвращают пути к директориям `tests/items/` и `tests/suites/`.

|`runs_root_dir()`
|Возвращает корневую директорию для результатов запусков `runs/`.

|`views_root_dir()` и специализированные функции
|Возвращают пути для кэшированных представлений: `views/leaderboard/`, `views/config_results/`, `views/test_results/`, `views/families/`.

|`config_item_path(config_id)`
|Возвращает путь к JSON-файлу конфигурации: `configs/items/{config_id}.json`.

|`family_path(family_id)`
|Возвращает путь к JSON-файлу семейства: `configs/families/{family_id}.json`.

|`test_item_path(test_id)`
|Возвращает путь к JSON-файлу теста: `tests/items/{test_id}.json`.

|`suite_path(suite_id)`
|Возвращает путь к JSON-файлу сьюта: `tests/suites/{suite_id}.json`.

|`run_month_dir(dt)`, `run_dir(run_id, dt)`
|Структурирует результаты запусков по месяцам: `runs/{YYYY-MM}/{run_id}/`.

|`run_manifest_path()`, `run_index_path()`, `run_results_dir()`, `result_path()`
|Построение путей внутри директории запуска: `run.json`, `index.json`, `results/{result_id}.json`.

|`configs_list_view_path()`, `tests_list_view_path()`, `suites_list_view_path()`, `runs_list_view_path()`
|Пути к кэш-файлам списков сущностей в `views/`.

|`leaderboard_view_path(test_id)`
|Путь к лидерборду для конкретного теста: `views/leaderboard/{test_id}.json`.

|`config_results_view_path(config_id)`, `test_results_view_path(test_id)`
|Пути к кэш-файлам результатов по конфигурации или тесту.

|`family_view_path(family_id)`
|Путь к кэш-файлу представления семейства.

|`board_layout_view_path()`
|Путь к файлу layout-настроек доски: `views/board_layout.json`.
|===

== Хранение конфигураций

Файл: `config_store.py`

Управляет конфигурациями и семействами конфигураций.

[cols="1,3"]
|===
|Функция |Описание

|`get_config(config_id)`
|Загружает конфигурацию по ID. Возвращает `dict` или `None`.

|`save_config(payload)`
|Сохраняет конфигурацию. `payload` должен содержать поле `"id"`.

|`delete_config(config_id)`
|Удаляет конфигурацию по ID.

|`list_configs()`
|Возвращает список всех конфигураций (читает все `*.json` из `configs/items/`).

|`get_family(family_id)`
|Загружает семейство конфигураций по ID.

|`save_family(payload)`
|Сохраняет семейство. `payload` должен содержать поле `"id"`.
|===

== Хранение запусков

Файл: `run_store.py`

Управляет результатами запусков. Каждый запуск хранится в отдельной директории с манифестом, индексом и файлами результатов.

[cols="1,3"]
|===
|Функция |Описание

|`save_run_manifest(run_id, payload, dt)`
|Сохраняет манифест запуска (`run.json`).

|`save_run_index(run_id, payload, dt)`
|Сохраняет индекс запуска (`index.json`).

|`save_result(run_id, result_id, payload, dt)`
|Сохраняет отдельный результат в `results/{result_id}.json`.

|`run_bundle(run_id, dt)`
|Возвращает полный бандл запуска: манифест, индекс и все результаты в виде списка.

|`list_runs()`
|Возвращает список всех запусков (объединяет манифест и индекс), отсортированный по дате (новые сначала).
|===

=== Структура директории запуска

Для каждого запуска создаётся структура:

[source]
----
runs/
└── 2025-06/
    └── {run_id}/
        ├── run.json           # манифест запуска
        ├── index.json         # индекс тестов/конфигураций
        └── results/
            ├── {result_id}.json
            └── ...
----

== Архитектурные заметки

. **Файловая персистентность**: Все данные хранятся в JSON-файлах, что упрощает отладку, миграцию и резервное копирование.

. **Изоляция хранилища**: Хранилище может находиться вне репозитория кода (настраивается через `BENCHFILE_STORAGE_ROOT`).

. **Кэширование представлений**: Для ускорения частых запросов создаются кэш-файлы в `views/`. Они обновляются соответствующими сервисами при изменениях данных.

. **Структурирование по времени**: Запуски группируются по месяцам для эффективного управления большими объёмами данных.

. **Автоматическое создание директорий**: Функция `ensure_parent()` гарантирует, что родительские директории будут созданы автоматически при записи файлов.

. **Сортировка**: Списки файлов и запусков всегда возвращаются в отсортированном виде для предсказуемого порядка.
