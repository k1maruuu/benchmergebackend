== Обзор

Этот документ описывает все Pydantic схемы, используемые в API BenchMerge Backend.
Схемы определяют структуру запросов и ответов для различных эндпоинтов системы.

== Файлы схем

=== board_layout.py - Схемы для управления раскладкой доски

==== BoardLayoutPosition

Определяет позицию элемента на доске (UI компонент).

|===
| Поле | Тип | Описание | Обязательное

| x
| float
| Координата X
| Да

| y
| float
| Координата Y
| Да
|===

[source,json]
----
{
  "x": 100.5,
  "y": 200.0
}
----

==== BoardLayoutPayload

Контейнер для позиций всех элементов на доске.

|===
| Поле | Тип | Описание | Обязательное

| positions
| Dict[str, BoardLayoutPosition]
| Словарь: ключ - ID элемента, значение - позиция
| Нет (по умолчанию: {})
|===

[source,json]
----
{
  "positions": {
    "element_1": {"x": 50.0, "y": 100.0},
    "element_2": {"x": 150.0, "y": 200.0}
  }
}
----

=== configs.py - Схемы для управления конфигурациями моделей

==== ConfigTrigger

Флаги автоматизации для конфигурации.

|===
| Поле | Тип | Описание | Обязательное | Значение по умолчанию

| collector
| bool
| Активировать Collector trigger для этой конфигурации
| Нет
| false
|===

[source,json]
----
{
  "collector": true
}
----

*Примечание:* Класс спроектирован как объект для возможности добавления новых флагов в будущем (например, improver).

==== ConfigCreate

Схема для создания новой конфигурации модели.

|===
| Поле | Тип | Описание | Обязательное | Значение по умолчанию

| config_id
| Optional[str]
| Уникальный ID конфигурации (генерируется backend, если не указан)
| Нет
| None

| name
| str
| Название конфигурации
| Да
| —

| model
| str
| Имя модели LLM
| Нет
| "gemma3:4b"

| service
| Optional[str]
| Диспетчер service/route (orchestrator, workshop, puml_gen и др.)
| Нет
| None

| node
| Optional[str]
| Опциональный LaunchPad node id (используется только для Pull/Push payload sync)
| Нет
| None

| system_prompt
| Optional[str]
| Системный промпт для модели
| Нет
| None

| options
| Dict[str, Any]
| Параметры генерации AI (temperature, top_p и др.)
| Нет
| {}

| description
| Optional[str]
| Описание конфигурации
| Нет
| None

| trigger
| Optional[ConfigTrigger \| bool \| Dict[str, Any]]
| Флаги автоматизации
| Нет
| None
|===

[source,json]
----
{
  "name": "Test Config",
  "model": "gemma3:4b",
  "service": "workshop",
  "node": "god_node",
  "system_prompt": "Ты помощник разработчика",
  "options": {
    "temperature": 0.7,
    "top_p": 0.9
  },
  "description": "Конфигурация для тестирования",
  "trigger": {
    "collector": true
  }
}
----

==== ConfigTriggerPatch

Запатчить флаги автоматизации для одной конфигурации.

*Поддерживаемые форматы запроса:*

[source,json]
----
// Формат 1: прямой ключ
{ "collector": true }

// Формат 2: вложенный trigger
{ "trigger": { "collector": true } }

// Формат 3: boolean
{ "trigger": true }
----

|===
| Поле | Тип | Описание |

| trigger
| Optional[ConfigTrigger \| bool \| Dict[str, Any]]
| Объект с флагами или boolean

| collector
| Optional[bool]
| Прямой флаг collector
|===

Метод `trigger_payload()` возвращает финальный Dict[str, Any] для применения к конфигурации.

==== ConfigOptionsPatch

Запатчить параметры генерации AI для конфигурации без удаления существующих ключей.

*Поддерживаемые форматы запроса:*

[source,json]
----
// Формат 1: прямой
{ "temperature": 0.3, "top_p": 0.9 }

// Формат 2: вложенный options
{ "options": { "temperature": 0.3, "top_p": 0.9 } }
----

|===
| Поле | Тип | Описание |

| options
| Optional[Dict[str, Any]]
| Параметры генерации
|===

Метод `options_payload()` возвращает финальный Dict[str, Any] для применения к конфигурации.

*Примечание:* Schema позволяет неизвестные ключи, так как разные LLM провайдеры имеют разные параметры. Валидация общих ключей происходит на уровне сервиса.

==== OffspringCreateRequest

Запрос для создания потомков конфигурации (используется для генерации вариаций).

|===
| Поле | Тип | Описание | Значение по умолчанию

| count
| int
| Количество потомков для создания
| 3

| name_prefix
| Optional[str]
| Префикс для имен потомков
| None

| mutation_strategy
| str
| Стратегия мутации
| "small_variation"
|===

[source,json]
----
{
  "count": 5,
  "name_prefix": "variant_",
  "mutation_strategy": "large_variation"
}
----

=== gateway.py - Схемы для прокси-запросов к Dispatcher

==== GatewayRunRequest

Запрос для запуска работы через Dispatcher.

|===
| Поле | Тип | Описание | Обязательное

| service
| str (ServiceName)
| Имя сервиса Dispatcher (получить через GET /gateway/connections)
| Да

| node
| Optional[str] (NodeName)
| ID узла из service.nodes выбранного сервиса
| Нет

| prompt
| Optional[str]
| Системный промпт для выбранного узла
| Нет
|===

*Валидация:*
- Если указан `node`, то обязателен `prompt`
- Если указан `prompt`, то обязателен `node`

[source,json]
----
{
  "service": "workshop",
  "node": "god_node",
  "prompt": "Ты разработчик по Rust"
}
----

==== LaunchpadSetSystemPromptRequest

Запрос для установки системного промпта (поддерживает два формата).

*Формат 1 - UI (рекомендуемый):*
[source,json]
----
{
  "service": "workshop",
  "node": "god_node",
  "prompt": "Ты разработчик по Rust"
}
----

*Формат 2 - Dispatcher-compatible (динамический):*
[source,json]
----
{
  "service": "workshop",
  "god_node": "Ты разработчик по Rust"
}
----

|===
| Поле | Тип | Описание | Обязательное

| service
| str (ServiceName)
| Имя сервиса Dispatcher
| Да

| node
| Optional[str] (NodeName)
| ID узла
| Нет

| prompt
| Optional[str]
| Системный промпт
| Нет
|===

*Валидация:* Передайте либо `node+prompt`, либо `service` + динамическое поле узла.

Метод `dispatcher_payload()` формирует финальный payload для отправки в Dispatcher.

==== GatewayRunResponse

Ответ от запуска работы через Gateway.

|===
| Поле | Тип | Описание

| service
| Optional[str]
| Имя сервиса

| node
| Optional[str]
| ID узла

| updated_prompt
| Optional[bool]
| Был ли обновлен промпт

| status
| Optional[str]
| Статус выполнения

| steps
| Optional[list[dict]]
| Шаги выполнения

| raw
| dict[str, Any]
| Сырые данные ответа
|===

[source,json]
----
{
  "service": "workshop",
  "node": "god_node",
  "updated_prompt": true,
  "status": "completed",
  "steps": [
    {"step": 1, "description": "Init"},
    {"step": 2, "description": "Generate"}
  ],
  "raw": {}
}
----

| Поле | Тип | Описание | Обязательное | Значение по умолчанию

| str

| Да

| Да

| bool

==== CollectorDispatchRequest

Запрос для отправки результатов сохраненного запуска в сервис `collector` Dispatcher.

[source,json]
----
{
  "run_id": "run_456",
  "force": true
}
----

[source,json]
----
{
  "parent_config_id": "config_123",
  "force": false
}
----

|===
| Поле | Тип | Описание | Обязательное | Значение по умолчанию

| run_id
| Optional[str]
| ID запуска (если frontend знает точный run)
| Нет
| None

| parent_config_id
| Optional[str]
| ID родительской конфигурации (backend найдет последний завершенный run для этой семьи)
| Нет
| None

| force
| bool
| Принудительно перезаписать
| Нет
| false
|===

*Примечание:* Используйте `run_id` для конкретного запуска или `parent_config_id` для автоматического выбора последнего запуска семьи.

=== tests.py - Схемы для управления тестовыми заданиями

==== TestCreate

Схема для создания нового теста.

|===
| Поле | Тип | Описание | Обязательное | Значение по умолчанию

| test_id
| Optional[str]
| Уникальный ID теста (backend-native)
| Нет
| None

| name
| Optional[str]
| Название теста
| Нет
| None (генерируется)

| system_prompt
| Optional[str]
| Системный промпт
| Нет
| None

| user_prompt
| Optional[str]
| Пользовательский промпт (обязательно через validatior)
| Да (после валидации)
| —

| id
| Optional[str]
| ID теста (frontend-compatible)
| Нет
| None

| input
| Optional[str]
| Входной промпт (frontend-compatible)
| Нет
| None

| checks
| Dict[str, Any]
| Проверки для теста
| Нет
| {}

| tags
| List[str]
| Теги теста
| Нет
| []
|===

*Валидация:*
- Если `test_id` не указан, но есть `id`, то `test_id = id`
- Если `user_prompt` не указан, но есть `input`, то `user_prompt = input`
- Если `name` не указан, генерируется из `test_id`, `id` или "Untitled test"
- `user_prompt` обязателен после нормализации

[source,json]
----
{
  "test_id": "test_123",
  "name": "Example Test",
  "system_prompt": "Ты эксперт",
  "user_prompt": "Решите задачу...",
  "checks": {
    "accuracy": true,
    "latency_threshold": 100
  },
  "tags": ["regression", "smoke"]
}
----

[source,json]
----
// Frontend-compatible формат
{
  "id": "test_456",
  "input": "Что такое Python?",
  "tags": ["qa"]
}
----

== Примечания

- Все схемы используют Pydantic v2 для валидации данных
- Опциональные поля могут быть `None` или иметь значения по умолчанию
- Схемы с `model_validator` могут изменять значения полей при валидации
- Некоторые схемы поддерживают несколько форматов запросов для совместимости с frontend и Dispatcher
- Поля типа `Dict[str, Any]` позволяют динамические значения без строгой типизации
